## Project Overview
A secure, web-based Patient Data Management System (PDMS)
designed for a small to medium-sized outpatient primary
healthcare clinic serving both acute and chronic care patients.

The system manages patient clinical data from registration
through diagnosis, treatment, and long-term chronic disease
monitoring. It is NOT a full Hospital Management System.

Primary audience: General practitioners, nurses, receptionists,
lab technicians, and patients in an outpatient clinic setting.

## Project Type
University IT Project
Focus: Software Architecture + Advanced Database Concepts
Timeline: 2 months
Presentation: Live demo with two patient scenarios

## 👥 System Actors and Roles

### 1. System Admin
- Manages user accounts (create, update, deactivate)
- Assigns roles and permissions to users
- Views and filters system audit logs
- Configures system roles and permissions

### 2. Doctor / Physician
- Searches and views full patient profiles and history
- Creates and closes clinical encounters
- Writes SOAP format clinical notes
- Records ICD-10 diagnoses (primary and secondary)
- Issues prescriptions (system checks allergies)
- Orders lab tests with priority levels
- Views lab results and imaging records
- Creates specialist referrals
- Creates and updates patient care plans
- Views trends in vitals and lab results over time

### 3. Nurse
- Records patient vitals per encounter
- Views patient profiles and encounter history
- Updates care and nursing notes
- Views and updates vaccination records

### 4. Receptionist
- Registers new patients
- Searches existing patients
- Books, reschedules, and cancels appointments
- Checks patients in on arrival

### 5. Lab Technician
- Views pending lab orders (sorted by priority)
- Uploads lab results with values and file attachments
- Flags abnormal results
- Updates lab order status

### 6. Patient
- Views own medical records and visit history
- Views prescriptions and lab results
- Books and cancels appointments
- Views vaccination history

## 🏥 Two Core Clinical Workflows

### Workflow 1: Acute Visit (New or Returning Patient)
Used for: Cold, flu, injury, infection, one-time consultation

Flow:
1.  Receptionist registers patient or searches existing record
2.  Receptionist books appointment
3.  Nurse records vitals on patient arrival
4.  Doctor opens encounter → create_encounter stored proc fires
5.  Doctor writes SOAP note
6.  Doctor records ICD-10 diagnosis
7.  Doctor issues prescription → allergy check runs
8.  Doctor orders lab test if needed
9.  Lab technician uploads result → abnormal flag set if needed
10. Doctor closes encounter
11. Patient views records via portal

### Workflow 2: Chronic Disease Follow-up
Used for: Diabetes, hypertension, asthma, ongoing conditions

Flow:
1.  Receptionist books scheduled follow-up appointment
2.  Nurse records vitals (system shows previous values)
3.  Doctor opens encounter
4.  Doctor reviews previous encounters, lab trends, care plan
5.  Doctor writes SOAP note referencing previous assessment
6.  Doctor orders periodic labs (e.g. quarterly HbA1c)
7.  Lab technician uploads results
8.  Doctor reviews results and trending data
9.  Doctor updates care plan goals and interventions
10. Doctor updates or renews prescriptions
11. Doctor creates referral if complication detected
12. Encounter closed, next follow-up scheduled

## 🗄️ Database Design

### Database Engine
Microsoft SQL Server (local development via SSMS)
Deployed on AWS EC2 (SQL Server on Windows or RDS)

### Connection Details (local development)
Server: localhost
Database: pdms_db
Auth: Windows Authentication (local)
      SQL Server Auth (EC2 production)

### Advanced Database Concepts Demonstrated
- Normalization: All 20 entities in 3NF/BCNF
- Indexing: On patient_id, encounter_date, doctor_id,
            is_abnormal, ordered_at, icd_code
- Stored Procedures: usp_create_encounter,
                     usp_record_prescription
- Triggers: trg_audit on all clinical tables,
            trg_calculate_bmi on vitals,
            trg_flag_abnormal_vitals on vitals
- ACID Transactions: All stored procs use
                     BEGIN/COMMIT/ROLLBACK TRANSACTION
- JSON Storage: audit_logs old_value and new_value
                stored as NVARCHAR(MAX) using FOR JSON
- Full-Text Search: On clinical_notes and diagnoses
                    using SQL Server FTS CONTAINS()
- Soft Delete: is_removed + valid_from + valid_until
- Three Timestamps: recorded_at, event_date, validated_at
- Redis Caching: Patient summaries and RBAC lookups

### SQL Server Data Types Used

| Concept          | SQL Server Type       | Notes                    |
|------------------|-----------------------|--------------------------|
| Primary Key      | UNIQUEIDENTIFIER      | DEFAULT NEWID()          |
| Short text       | NVARCHAR(n)           | e.g. NVARCHAR(100)       |
| Long text        | NVARCHAR(MAX)         | replaces TEXT            |
| Boolean          | BIT                   | 1=true, 0=false          |
| Timestamp        | DATETIME2             | replaces TIMESTAMP       |
| Auto-increment   | INT IDENTITY(1,1)     | replaces SERIAL          |
| JSON data        | NVARCHAR(MAX)         | validate with ISJSON()   |
| Decimal numbers  | DECIMAL(10,2)         | for BMI, measurements    |

### Stored Procedures (T-SQL)

#### usp_create_encounter
CREATE PROCEDURE usp_create_encounter
    @p_patient_id       UNIQUEIDENTIFIER,
    @p_doctor_id        UNIQUEIDENTIFIER,
    @p_appointment_id   UNIQUEIDENTIFIER = NULL,
    @p_encounter_type   NVARCHAR(20),
    @p_chief_complaint  NVARCHAR(MAX),
    @p_created_by       UNIQUEIDENTIFIER,
    @p_encounter_id     UNIQUEIDENTIFIER OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    BEGIN TRANSACTION;
    BEGIN TRY

        -- Step 1: Generate new encounter ID
        SET @p_encounter_id = NEWID();

        -- Step 2: Get visit number for this patient
        DECLARE @visit_number INT;
        SELECT @visit_number = ISNULL(COUNT(*), 0) + 1
        FROM encounters
        WHERE patient_id = @p_patient_id;

        -- Step 3: Insert encounter
        INSERT INTO encounters (
            encounter_id, patient_id, doctor_id,
            appointment_id, encounter_type,
            chief_complaint, status,
            visit_number, created_at
        )
        VALUES (
            @p_encounter_id, @p_patient_id, @p_doctor_id,
            @p_appointment_id, @p_encounter_type,
            @p_chief_complaint, 'open',
            @visit_number, GETDATE()
        );

        -- Step 4: Update appointment status if provided
        IF @p_appointment_id IS NOT NULL
        BEGIN
            UPDATE appointments
            SET status = 'completed'
            WHERE appointment_id = @p_appointment_id
            AND status != 'cancelled';
        END

        -- Step 5: Insert empty SOAP note shell
        INSERT INTO clinical_notes (
            note_id, encounter_id, doctor_id, recorded_at
        )
        VALUES (
            NEWID(), @p_encounter_id,
            @p_doctor_id, GETDATE()
        );

        -- Step 6: Insert audit log
        INSERT INTO audit_logs (
            log_id, user_id, action,
            table_affected, record_id, timestamp
        )
        VALUES (
            NEWID(), @p_created_by, 'CREATE',
            'encounters', @p_encounter_id, GETDATE()
        );

        COMMIT TRANSACTION;

    END TRY
    BEGIN CATCH
        ROLLBACK TRANSACTION;
        THROW;
    END CATCH
END;

#### usp_record_prescription
CREATE PROCEDURE usp_record_prescription
    @p_encounter_id   UNIQUEIDENTIFIER,
    @p_patient_id     UNIQUEIDENTIFIER,
    @p_doctor_id      UNIQUEIDENTIFIER,
    @p_drug_name      NVARCHAR(200),
    @p_dosage         NVARCHAR(50),
    @p_frequency      NVARCHAR(50),
    @p_duration       NVARCHAR(50),
    @p_route          NVARCHAR(30),
    @p_instructions   NVARCHAR(MAX),
    @p_prescription_id UNIQUEIDENTIFIER OUTPUT
AS
BEGIN
    SET NOCOUNT ON;
    BEGIN TRANSACTION;
    BEGIN TRY

        -- Step 1: Check encounter is open
        IF NOT EXISTS (
            SELECT 1 FROM encounters
            WHERE encounter_id = @p_encounter_id
            AND status = 'open'
        )
        BEGIN
            RAISERROR('ENCOUNTER_CLOSED: Cannot prescribe
                       on a closed encounter.', 16, 1);
            RETURN;
        END

        -- Step 2: Check patient allergies
        IF EXISTS (
            SELECT 1 FROM allergies
            WHERE patient_id = @p_patient_id
            AND is_removed = 0
            AND LOWER(allergen) = LOWER(@p_drug_name)
        )
        BEGIN
            RAISERROR('ALLERGY_CONFLICT: Patient is allergic
                       to %s.', 16, 1, @p_drug_name);
            RETURN;
        END

        -- Step 3: Insert prescription
        SET @p_prescription_id = NEWID();

        INSERT INTO prescriptions (
            prescription_id, encounter_id, patient_id,
            doctor_id, drug_name, dosage, frequency,
            duration, route, instructions,
            is_active, issued_at
        )
        VALUES (
            @p_prescription_id, @p_encounter_id,
            @p_patient_id, @p_doctor_id,
            @p_drug_name, @p_dosage, @p_frequency,
            @p_duration, @p_route, @p_instructions,
            1, GETDATE()
        );

        -- Step 4: Insert audit log
        INSERT INTO audit_logs (
            log_id, user_id, action,
            table_affected, record_id, timestamp
        )
        VALUES (
            NEWID(), @p_doctor_id, 'CREATE',
            'prescriptions', @p_prescription_id, GETDATE()
        );

        COMMIT TRANSACTION;

    END TRY
    BEGIN CATCH
        ROLLBACK TRANSACTION;
        THROW;
    END CATCH
END;

### Triggers (T-SQL)

#### trg_audit_encounters (template — repeat for all tables)
CREATE TRIGGER trg_audit_encounters
ON encounters
AFTER INSERT, UPDATE, DELETE
AS
BEGIN
    SET NOCOUNT ON;

    DECLARE @action NVARCHAR(10);

    IF EXISTS (SELECT 1 FROM inserted)
       AND EXISTS (SELECT 1 FROM deleted)
        SET @action = 'UPDATE';
    ELSE IF EXISTS (SELECT 1 FROM inserted)
        SET @action = 'CREATE';
    ELSE
        SET @action = 'DELETE';

    INSERT INTO audit_logs (
        log_id, action, table_affected,
        old_value, new_value, timestamp
    )
    SELECT
        NEWID(),
        @action,
        'encounters',
        (SELECT * FROM deleted FOR JSON PATH),
        (SELECT * FROM inserted FOR JSON PATH),
        GETDATE();
END;

-- Apply same trigger pattern to:
-- clinical_notes, diagnoses, vitals, prescriptions,
-- lab_orders, lab_results, imaging_records,
-- medical_history, allergies, vaccinations,
-- referrals, care_plans


#### trg_calculate_bmi
CREATE TRIGGER trg_calculate_bmi
ON vitals
AFTER INSERT, UPDATE
AS
BEGIN
    SET NOCOUNT ON;

    UPDATE v
    SET bmi = ROUND(
        i.weight_kg / POWER(i.height_cm / 100.0, 2), 2
    )
    FROM vitals v
    INNER JOIN inserted i ON v.vital_id = i.vital_id
    WHERE i.weight_kg IS NOT NULL
    AND i.height_cm IS NOT NULL
    AND i.height_cm > 0;
END;


#### trg_flag_abnormal_vitals
CREATE TRIGGER trg_flag_abnormal_vitals
ON vitals
AFTER INSERT
AS
BEGIN
    SET NOCOUNT ON;

    UPDATE v
    SET is_abnormal = 1
    FROM vitals v
    INNER JOIN inserted i ON v.vital_id = i.vital_id
    WHERE i.heart_rate > 100
       OR i.heart_rate < 60
       OR i.blood_pressure_sys > 140
       OR i.blood_pressure_sys < 90
       OR i.blood_pressure_dia > 90
       OR i.oxygen_saturation < 95
       OR i.temperature > 37.5
       OR i.temperature < 36.0
       OR i.respiratory_rate > 20
       OR i.respiratory_rate < 12;
END;

## 🔐 Security Design

### Authentication
- JWT tokens (ACCESS_TOKEN_EXPIRE_MINUTES = 30)
- Refresh token pattern
- bcrypt password hashing (cost factor = 12)
- Rate limiting: max 5 failed logins → 15 min lockout

### Authorization
- RBAC enforced at two levels:
  1. API layer: require_permission() FastAPI dependency
  2. Business logic layer: service-level rule checks
- Permission check: user.role.permissions contains name

### Audit
- All data modifications captured by SQL Server triggers
- All VIEW actions captured by application middleware
- Audit logs are immutable — no UPDATE or DELETE ever
- Every request logs: user_id, action, ip, timestamp

### Data Integrity
- All FKs enforced at DB level with FOREIGN KEY constraints
- NOT NULL constraints on all critical fields
- UNIQUE constraints on mrn, username, email
- CHECK constraints on status and enum-like fields
- ACID transactions in all stored procedures
- Soft delete pattern using is_removed BIT flag

## 🔑 Business Rules (enforce in service layer)

1.  Prescription CANNOT be issued on a closed encounter
2.  Prescription BLOCKED if patient has matching allergy
3.  Lab result CANNOT be uploaded without matching order
4.  Encounter CANNOT be created for cancelled appointment
5.  Only ordering doctor can close an encounter
6.  Patient MRN unique — format PDMS-YYYY-NNNNN
7.  Audit logs NEVER deleted or modified
8.  User can only have ONE role at a time
9.  Vitals MUST link to an open encounter
10. Lab order status forward only:
    ordered → in-progress → completed
11. Soft delete for allergies and prescriptions
    use is_removed = 1, never hard DELETE
12. is_chronic = 1 means condition has no end date
13. BMI auto-calculated by trg_calculate_bmi trigger
14. Vitals is_abnormal auto-set by trigger on insert
15. visit_number auto-incremented per patient
    calculated in usp_create_encounter
